Asegura tu web de ataques

🗓️Septiembre 07, 2020🔒 Seguridad💻Informatica

Para muchas empresas, 2020 fue todo un cambio de trabajo remoto en los sistemas empresariales basados en la nube, y los equipos de seguridad de las aplicaciones tuvieron que adaptarse a un cambio de uso y a un número creciente de desafíos.

Las vulnerabilidades de las aplicaciones web fueron la causa del 43% de las violaciones de datos en 2019, según The 2020 Verizon Data Breach Investigations Report. Sorprendentemente, el 79% de las organizaciones empujaron intencionalmente código vulnerable a la producción, al mismo tiempo que consideraban que su propia postura de seguridad de aplicaciones era superior a 7 de 10, según la investigación de Enterprise Strategy Group.

Dado que el coste medio de las filtraciones de datos es de 3,86 millones de dólares, la seguridad de las aplicaciones no es algo que las empresas puedan ignorar. Las cifras van en aumento: un 12% en los últimos cinco años.

En comparación con otros activos informáticos, las aplicaciones web son especialmente vulnerables a los ataques, porque están expuestas a Internet. Muchos vectores de ataque contra las aplicaciones web se centran en la manipulación de las entradas de los usuarios a través de los formularios web, y de las entradas de las máquinas a través de las API. En este artículo, hablaremos de las vulnerabilidades de las aplicaciones web y de las mejores prácticas para protegerlas contra ataques maliciosos y daños accidentales.

Vulnerabilidades de las aplicaciones web Las vulnerabilidades de las aplicaciones web son puntos débiles de seguridad que permiten a los actores de la amenaza manipular el código fuente, obtener acceso no autorizado, robar datos o interferir de otro modo en el funcionamiento normal de la aplicación.

El documento OWASP Top 10 enumera los riesgos de seguridad más críticos para las aplicaciones web. Repasemos algunos de los vectores de ataque más conocidos:

Inyección SQL: se produce cuando los atacantes utilizan código SQL malicioso para manipular las bases de datos del backend. El resultado puede incluir el listado de datos no autorizado, la caída (eliminación) de tablas y el acceso administrativo no autorizado.

Cross-Site Scripting (XSS)-un ataque dirigido a los usuarios de una aplicación. Puede utilizarse para acceder a cuentas de usuario, inyectar troyanos o cambiar el contenido de la página para engañar a los usuarios o desfigurar un sitio web. Otra variante más peligrosa es el XSS almacenado, cuando el código malicioso se inyecta de forma persistente en la aplicación. El XSS reflejado es cuando los scripts maliciosos se reflejan desde la aplicación al navegador del usuario.

Inclusión remota de archivos (RFI): inyección remota de archivos en el servidor de una aplicación web. Esto puede llevar a la ejecución de scripts y código malicioso en las aplicaciones, a comprometer el servidor web y al robo de datos.

Cross-Site Request Forgery (CSRF): un ataque que puede provocar transferencias de fondos no deseadas, cambios de contraseña o robo de datos. Implica que un atacante aproveche la sesión abierta de un usuario, haciendo que el navegador del usuario realice acciones sin saberlo en un sitio en el que el usuario ha iniciado la sesión.

La desinfección de las entradas y salidas de la aplicación, por ejemplo con la libreria Dompurify, y la adopción de prácticas de codificación seguras pueden proteger las aplicaciones contra la mayoría de las vulnerabilidades. Sin embargo, esto no es suficiente. Las aplicaciones web están en constante desarrollo, y las pruebas de seguridad deben incorporarse en cada etapa del ciclo de vida del desarrollo, para identificar y corregir el código vulnerable desde el principio.

Además, la mayoría de las aplicaciones web utilizan componentes de código abierto de terceros, que pueden ser a su vez vulnerables, y deben ser escaneados de forma continua. Técnicas y herramientas de seguridad de las aplicaciones web

La técnica de seguridad más poderosa en el desarrollo web es pensar en la seguridad en todos los detalles, incluso los más pequeños. La seguridad de las aplicaciones web no sólo se refiere a la base de código, sino también a los procesos del proyecto, el almacenamiento de datos, las políticas de la empresa, etc. Al final, no importa lo bueno y protegido que sea tu código, si tienes la documentación de tu API a disposición del público, o la contraseña de súper administrador escrita en una pizarra en la oficina. Sergey Ivanov, jefe de control de calidad en MobiDev.

Así que, en primer lugar, la seguridad en las aplicaciones proviene de la seguridad en el propio proyecto. Pero cuando todos los procesos son perfectos, ¿qué es lo siguiente?

Para obtener información sobre la seguridad de las aplicaciones web, en general, hay dos enfoques principales. Se trata de las pruebas de seguridad estáticas y dinámicas. No se excluyen el uno al otro, sino que deberían complementarse.

A continuación, le presentamos algunas tecnologías que puede utilizar para proteger sus aplicaciones web contra las vulnerabilidades, así como para responder a los ataques si se producen.

SAST

Las soluciones de pruebas estáticas de seguridad de aplicaciones (SAST) analizan el código fuente en busca de vulnerabilidades y riesgos de seguridad. Muchas aplicaciones web integran el escaneo de código en múltiples etapas de desarrollo, principalmente cuando se compromete el nuevo código a la base de código, y durante una construcción.

El SAST suele estar basado en reglas y los resultados del escaneo suelen incluir falsos positivos, por lo que tendrá que analizar y filtrar cuidadosamente los resultados para identificar los verdaderos problemas de seguridad. Nosotros utilizamos la herramienta de análisis estático SonarQube para supervisar los problemas de seguridad que puedan introducirse durante el desarrollo. Se recomienda integrarla con el pipeline CI/CD para que analice cada commit/merge commit. SonarQube tiene una buena representación visual y comprueba no sólo los aspectos de seguridad, sino también la mantenibilidad y la fiabilidad del código base. Soporta más de 20 lenguajes de programación diferentes, por lo que funciona para la mayoría de los frameworks de frontend y backend.

DAST

Las pruebas dinámicas de seguridad de las aplicaciones (DAST) consisten en probar el código desplegado o en ejecución para encontrar vulnerabilidades. Se puede realizar tanto de forma manual como automática, utilizando herramientas especiales.

Las pruebas manuales giran en torno al trabajo con la API de la aplicación con herramientas como Burp Suite, Fiddler, Postman. Las herramientas DAST de automatización envían un gran número de peticiones al código de la aplicación, incluyendo entradas inesperadas y maliciosas, en busca de vulnerabilidades. Analiza los resultados e identifica las debilidades de seguridad.

Después de un análisis de seguridad manual y concienzudo, utilizamos OwaspZap, un escáner de seguridad de aplicaciones web de código abierto, para acelerar las pruebas de regresión. Los escáneres no pueden reemplazar a los humanos en términos de creatividad, análisis de la causa raíz, o la capacidad de pensar fuera de la caja, pero pueden manejar las tareas de rutina a un ritmo y volumen mucho más rápido.

Pruebas de penetración

Las pruebas de penetración son una técnica de seguridad que combina herramientas de escaneo dinámicas con expertos en seguridad humanos para encontrar lagunas en la postura de seguridad de una aplicación web. Los pentesters actúan como verdaderos actores de amenazas, explotando vulnerabilidades, obteniendo acceso no autorizado, robando datos e interrumpiendo servicios. Sin embargo, lo hacen bajo contrato con el propietario de la aplicación web, bajo un alcance acordado, y sin causar un daño real a la organización. En comparación con SAST y DAST, esta técnica es más compleja de llevar a cabo, pero puede identificar riesgos adicionales que las herramientas automatizadas pueden pasar por alto.

XDR

Las soluciones de detección y respuesta ampliadas (XDR) son una nueva generación de plataformas de seguridad que ofrecen a los equipos de seguridad una interfaz que les permite detectar y responder a las amenazas dondequiera que existan en el entorno de TI.

XDR recopila datos de seguridad de todas las capas de la pila de seguridad, incluidas las aplicaciones web, las redes, las nubes privadas y públicas y los puntos finales. Aplica análisis avanzados y automatización para analizar, clasificar y detectar tanto las amenazas conocidas como las desconocidas. Lo más importante es que se integra directamente con las herramientas de seguridad y puede responder automáticamente a las amenazas en tiempo real.

Mejores prácticas de seguridad de las aplicaciones web en 2021 A continuación se presentan algunas de las mejores prácticas que puede utilizar para mejorar la seguridad de sus aplicaciones web.

Autenticación y control de acceso Aunque parezca obvio, muchas aplicaciones web no implementan medidas básicas de control de acceso. Asegúrese de seguir estos principios:

Aplique contraseñas seguras: utilice una recuperación de contraseñas segura, establezca políticas sensatas de caducidad y rotación de contraseñas y, preferiblemente, utilice la autenticación multifactorial.

Forzar la reautenticación cuando se acceda a funciones sensibles o se realicen transacciones.

Utilizar el principio de mínimo privilegio (POLP), y dar a cada usuario sólo los privilegios que necesita para desempeñar su función en el sistema.

Utilice SSL y encriptación y asegúrese de que las contraseñas y credenciales estén siempre encriptadas, tanto en reposo como en tránsito.

Supervise las cuentas de usuario y bloquee a los usuarios o solicite un cambio de contraseña si detecta actividad sospechosa.

Para las empresas, que además deben proporcionar seguridad de los datos y un estricto cumplimiento legal de leyes como GDPR o HIPAA, se necesita una solución avanzada.

Evite las malas configuraciones de seguridad Cualquiera que sea el CMS o el marco de desarrollo web que esté utilizando, hay muchas posibilidades de desconfiguración. Esté atento a los siguientes problemas:

Establezca siempre contraseñas de administrador seguras y cambie los nombres de usuario por defecto.

Proteja los archivos y directorios con configuración o contenido sensible.

No deje puertos abiertos si no necesita que estén abiertos innecesariamente

Actualice regularmente a la última versión estable todas las librerías de software, plugins y el propio framework.

Analice regularmente todos sus paquetes en busca de vulnerabilidades de seguridad.

Esté atento a las vulnerabilidades de seguridad y a las actualizaciones que afectan a su software e infraestructura.

Utilice protocolos de comunicación y de red seguros.

Asegúrese de que los certificados digitales estén actualizados.

Utilizamos Local PHP Security Checker después de cada actualización del compositor de PHP. Nos ayuda a evitar el uso de bibliotecas con vulnerabilidades conocidas. Normalmente, las actualizaciones regulares a la última versión estable resuelven muchos problemas de seguridad.

El uso de librerías y componentes de código abierto en el desarrollo de software es casi omnipresente, ya que alrededor del 99% de las aplicaciones tienen al menos un componente de código abierto, según el Informe de análisis de riesgos y seguridad de código abierto de 2020.

Algunas industrias, como el comercio minorista, la sanidad y la educación, experimentaron un crecimiento exponencial de los ingresos durante el año 2020, en gran parte debido a los cambios en el comportamiento de los consumidores y la interacción social durante el COVID. Como estas industrias utilizaron más código abierto en sus aplicaciones, tuvieron el mayor número de vulnerabilidades y vulnerabilidades de alto riesgo. Determinar qué componentes de código abierto son seguros debería ser una preocupación primordial para cualquier grupo de seguridad de aplicaciones.

Gestión de excepciones

Las excepciones son un aspecto de la seguridad de las aplicaciones web que se suele pasar por alto. Es común ver excepciones o errores mostrando largas trazas de pila al usuario-esta información es extremadamente valiosa para los atacantes. Nunca debe mostrar nada al usuario que no sea un mensaje de error que explique qué ha ido mal y qué puede hacer para resolverlo.

Asegúrate de planificar tu aplicación web para los escenarios "felices" y "no felices" de cada operación del usuario. Anticipe todos los errores posibles y manéjelos con excepciones significativas. Esto evitará que los atacantes exploten los casos límite para causar un comportamiento inesperado.

Gestionar cuidadosamente los contenedores

Muchas aplicaciones web se ejecutan en contenedores, utilizando Docker. Los contenedores pueden crear importantes problemas de seguridad si no se gestionan correctamente. Tenga en cuenta las siguientes directrices de seguridad:

Imágenes de confianza-cuando cree un contenedor, utilice siempre imágenes base de confianza, y analice las imágenes en busca de vulnerabilidades (incluso sus propias imágenes) antes de utilizarlas.

Utiliza secretos-es una muy mala práctica guardar credenciales u otra información sensible directamente en una imagen de contenedor porque estará disponible abiertamente en cualquier contenedor creado a partir de esa imagen. En su lugar, utilice el mecanismo de secretos de Docker o Kubernetes para almacenar la información sensible.

No conceda nunca acceso a la raíz: un contenedor que tenga acceso a la raíz en un sistema también concederá acceso a la raíz a un atacante, en caso de que el contenedor se vea comprometido. Defina siempre un usuario en sus imágenes y evite utilizar opciones como "contenedor privilegiado" en Kubernetes.

Segmentación de la red: asegúrese de que los contenedores sólo puedan acceder a otros sistemas si realmente lo necesitan. Ejecute los contenedores en una subred protegida y evite exponerlos a Internet a menos que sea absolutamente necesario.

El servicio Amazon Elastic Container Registry (ECR) proporciona escaneos de seguridad para las imágenes Docker. Es una característica bastante útil para entender qué problemas potenciales de seguridad tiene en sus imágenes Docker privadas.

Garantía de calidad y pruebas

Las pruebas de seguridad son esenciales para la seguridad de las aplicaciones web. Siga estas prácticas recomendadas:

Utilice el escaneo estático y dinámico: analice su código fuente en busca de vulnerabilidades durante el desarrollo utilizando las pruebas de seguridad de aplicaciones estáticas (SAST) y en producción utilizando las pruebas de seguridad de aplicaciones dinámicas (DAST).

Utilice pruebas de penetración: puede utilizar soluciones ligeras de pruebas de penetración como servicio (PTaaS) o, en el caso de aplicaciones a gran escala, una prueba de penetración periódica a gran escala realizada por un hacker ético certificado.

Adopte CI/CD: cada vez que actualice su aplicación, ejecute su código a través de un proceso de prueba automatizado y despliéguelo automáticamente, para asegurarse de que no introduce riesgos de seguridad debido a problemas de instalación.

Tenga en cuenta el cumplimiento de la normativa: la mayoría de las organizaciones del mundo están sujetas a la normativa GDPR de la Unión Europea. Las aplicaciones que procesan datos de tarjetas de crédito están sujetas a la norma PCI/DSS. Compruebe si existen normas o reglamentos de cumplimiento adicionales que afecten a su aplicación y aplique las medidas necesarias.

En la mayoría de los casos, nuestro proceso habitual de despliegue en CI/CD incluye los siguientes 5 pasos:

Escanear el código fuente en busca de violaciones de estilo de código Analizar las librerías de terceros utilizadas en busca de problemas de seguridad Ejecutar pruebas unitarias Preparar el nuevo código fuente para el despliegue. Crear y desplegar nuevas imágenes Docker en un entorno seleccionado. Todas las credenciales sensibles necesarias para ejecutar su aplicación web se guardan en el almacenamiento seguro del CI/CD. El despliegue se detendrá si alguno de estos pasos falla. Esto garantiza actualizaciones estables y un entorno estable y seguro.
